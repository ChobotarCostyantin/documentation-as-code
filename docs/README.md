# Документація проєкту betterstack

Система betterstack слугує єдиним центром для пошуку та оцінки програмного забезпечення. Її мета - допомогти спеціалістам приймати обґрунтовані рішення щодо вибору технологій на основі структурованих відгуків та порівняльних характеристик.

## Навігація по документації
- [API Contract](api-contract.md) - Опис специфікації та документування API як контракту.
- [System Specification (SSD)](architecture/SSD.md) - Опис меж системи, функціональних та нефункціональних вимог.
- [Software Design (SDD)](architecture/SDD.md) - Опис клієнт-серверної архітектури, структури модулів NestJS та Next.js.
- [Infrastructure (ISD)](architecture/ISD.md) - Опис Docker-середовища, налаштувань бази даних PostgreSQL та деплою.
- [CI/CD](ci-cd-and-dod/ci-cd-docs.md) - Опис автоматизації документації в CI/CD.
- [Definition of Done (DoD)](ci-cd-and-dod/definition-of-done.md) - Формалізація Definition of Done для документації.
- [Test Strategy](quality/test-strategy.md) - Загальний підхід до тестування (Unit, E2E для API, UI-тестування).
- [Traceability Matrix](quality/traceability-matrix.md) - Матриця зв'язків між вимогами (SSD) та конкретними тест-кейсами.

## Single Source of Truth (Єдине джерело істини)
Усі функціональні та нефункціональні вимоги описуються виключно в SSD. Він є єдиним джерелом істини для вимог. Документи із забезпечення якості (Test Strategy та Traceability Matrix) не вигадують нові правила, а жорстко посилаються на ідентифікатори вимог (FR-01, FR-02) з SSD. Архітектурні рішення (SDD) спираються на вимоги продуктивності з SSD.

## Версійність та актуалізація
Документація зберігається у Git разом із кодом, тому версіонується через Git Tags (наприклад, v1.0.0). Версія документації v1.0 відповідає стабільному релізу системи. При будь-якій зміні API або бізнес-вимог, спочатку вноситься зміна у відповідний Markdown файл (Pull Request), і лише після цього розробляється код. При значних змінах випускається нова версія документації (v1.1).
