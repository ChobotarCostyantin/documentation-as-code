# Definition of Done (Documentation)

Документ визначає обов'язкові критерії готовності змін у проєкті з точки зору документації. Зміна вважається завершеною лише після виконання всіх відповідних пунктів.

---

## Типи змін та вимоги

| Тип зміни                                       | Обов'язкові дії                                               | Перевірка                 |
|-------------------------------------------------|---------------------------------------------------------------|---------------------------|
| Додано новий API-ендпоінт                       | Оновити `openapi.yaml` (новий `path`, схеми запиту/відповіді) | `redocly lint` у CI       |
| Змінено request/response схему                  | Оновити відповідний `schema` у `components/schemas`           | `redocly lint` у CI       |
| Breaking change в API                           | Оновити `openapi.yaml` + збільшити major версію документації  | Code review + version tag |
| Додано новий UI-компонент                       | Створити `.stories.tsx` з базовим та альтернативним станом    | Storybook build у CI      |
| Змінено поведінку UI-компонента                 | Оновити відповідну Story (стани, props, description)          | Storybook build у CI      |
| Змінено архітектуру системи                     | Оновити відповідний документ (`SDD.md`, `ISD.md`, `SSD.md`)   | Code review               |
| Додано нову функціональну вимогу                | Додати запис у `SSD.md` + рядок у `traceability-matrix.md`    | Code review               |
| Нефункціональні зміни (рефакторинг, залежності) | Оновити `README.md` якщо змінився спосіб запуску              | Code review               |

---

## Перевірювані критерії готовності

### 1. Зміна API
- `openapi.yaml` містить новий або оновлений ендпоінт
- Описані всі коди відповіді (включаючи помилки `401`, `403`, `404`)
- Команда `redocly lint docs/openapi.yaml` завершується без помилок
- `api-contract.md` оновлений відповідно до змін

### 2. Зміна UI-компонента
- Існує файл `<ComponentName>.stories.tsx` поруч з компонентом
- Story містить мінімум два стани: базовий (`Default`) та альтернативний (`Disabled` / `Loading` / `Error`)
- Команда `npm run build-storybook` завершується без помилок

### 3. Breaking change
- Версія у `openapi.yaml` (`info.version`) збільшена по major (`v1.x` → `v2.0`)
- Створено Git Tag з новою версією
- Попередня версія документації збережена у GitHub Releases

### 4. Будь-яка зміна
- Гілка змерджена через Pull Request
- CI pipeline пройшов без помилок (`validate` job — зелений)
- Документація опублікована на GitHub Pages після merge у `main`

---

## Автоматизовані перевірки у CI

```
PR у main
    │
    ├── redocly lint docs/openapi.yaml
    ├── markdownlint docs/**/*.md
    └── npm run build-storybook

    ↓ все зелене → дозволено merge
```

## Приклад застосування DoD

**Сценарій:** розробник додає ендпоінт `POST /api/v1/software/{slug}/screenshots`.

Чекліст перед merge:

- [ ] Ендпоінт описаний у `openapi.yaml` зі схемою тіла запиту та відповідями `201`, `401`, `404`

- [ ] `api-contract.md` оновлений — новий рядок у таблиці розділу Software

- [ ] `redocly lint` проходить без помилок

- [ ] CI pipeline зелений
